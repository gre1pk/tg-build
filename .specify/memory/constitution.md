# Конституция tg-build

## Основные принципы

### I. Нативный опыт Telegram

Каждая фича ДОЛЖНА ощущаться нативно внутри Telegram Mini App: использовать
`@telegram-apps/sdk-react` и `@telegram-apps/telegram-ui`, уважать параметры темы
Telegram и оптимизировать под мобильное touch-взаимодействие.

- Новые экраны НЕ ДОЛЖНЫ ломать навигацию и конвенции темизации Mini App.
- Авторизация ДОЛЖНА проходить через валидированный `initData` на сервере и JWT-сессии;
  клиентский код НЕ ДОЛЖЕН обходить auth в production-путях.
- Основные пользовательские потоки (главная → ткани → заявка → контакт с мастером)
  ДОЛЖНЫ оставаться короткими и доступными без глубокой вложенности.

**Обоснование**: пользователи открывают приложение из Telegram на телефоне; лишнее
трение или ненативный UI снижают доверие и конверсию в выбор ткани и заявку.

### II. UX с приоритетом фото

Визуальное доказательство качества — ключ к конверсии для студии перетяжки. Изображения
ДОЛЖНЫ быть заметными, а не декоративными.

- Каталог и карточка ткани ДОЛЖНЫ показывать крупные, читаемые по фактуре образцы —
  не мелкие превью (минимальная область ДОЛЖНА позволять оценить текстуру материала).
- Портфолио и контент «до/после» ДОЛЖНЫ использовать реальные фото работ, когда они есть.
- Копирайт и вёрстка ДОЛЖНЫ поддерживать процесс: фото мебели → выбор ткани → перетяжка →
  связь с мастером.

**Обоснование**: клиенты выбирают ткань по текстуре и качеству; мелкие превью и стоковые
фото подрывают обещание продукта из `PRODUCT.md`.

### III. Паритет mock / live

Разработка и production ДОЛЖНЫ разделять одни контракты и границы поведения.

- Весь доступ к данным ДОЛЖЕН идти через слой репозитория (`createRepository`,
  `mockRepository`, `apiRepository`) с общими TypeScript-типами в `src/data/`.
- Фичи ДОЛЖНЫ работать в `npm run dev:mock` до того, как потребуются live-секреты API.
- Формы API-маршрутов (`/api/fabrics`, `/api/auth/*`) ДОЛЖНЫ оставаться согласованными
  между mock и live; расхождения ДОЛЖНЫ быть задокументированы в plan.md.
- В dev по умолчанию: mock-auth (`/api/auth/dev`) + live API к Supabase — допустимый режим.

**Обоснование**: mock ускоряет локальную итерацию; паритет предотвращает сюрпризы при
деплое на Vercel.

### IV. Безопасность на сервере (ОБЯЗАТЕЛЬНО)

Секреты и границы доверия ДОЛЖНЫ оставаться на сервере.

- `TELEGRAM_BOT_TOKEN`, `AUTH_JWT_SECRET`, `SUPABASE_SERVICE_ROLE_KEY` и прочие секреты БД
  НИКОГДА НЕ ДОЛЖНЫ попадать в клиентские бандлы или переменные `VITE_*`.
- HMAC-валидация Telegram `initData` ДОЛЖНА выполняться на сервере
  (`api/lib/validateInitData.js`) до выдачи сессии.
- Новые защищённые эндпоинты ДОЛЖНЫ проверять JWT и возвращать корректные HTTP-коды.
- Админ-маршруты (`/api/admin/*`, UI `/admin/*`) ДОЛЖНЫ ограничивать доступ списком
  `ADMIN_TELEGRAM_IDS`.
- `/api/auth/dev` ДОЛЖЕН быть доступен только локально (не на Vercel).

**Обоснование**: Mini App работает в недоверенном клиенте; серверная валидация — единственная
безопасная граница auth.

### V. Простота и инкрементальная поставка

Предпочитайте минимальное изменение, дающее ценность пользователю. YAGNI, если в
`PRODUCT.md` или `CHECKLIST.md` явно не требуется больше.

- Фичи ДОЛЖНЫ нарезаться на независимо тестируемые user stories (P1 → P2 → P3).
- Новые зависимости, абстракции и инфраструктура ДОЛЖНЫ быть обоснованы в таблице
  Complexity Tracking плана, если нарушают этот принцип.
- Стили ДОЛЖНЫ переиспользовать SCSS modules (`src/ui/`, `src/styles/`) и токены из
  `DESIGN.md` до введения параллельных дизайн-систем.

**Обоснование**: приложение студии на малой команде выигрывает от быстрой поставки каталога
и заявок, а не от ранней архитектуры.

## Язык документации

Все артефакты Spec Kit (`/speckit-specify`, `/speckit-plan`, `/speckit-tasks` и связанные
команды) ДОЛЖНЫ быть на **русском**:

- spec, plan, tasks, research, checklists в `specs/`
- правки конституции и governance-текст в этом файле

На **английском** остаются: исходный код, идентификаторы, пути файлов, имена env, маршруты
API, имена веток Git, npm-скрипты.

Пользовательский копирайт в спеках (подписи UI, ошибки, CTA) — **русский**, в духе
`PRODUCT.md`.

**Обоснование**: команда и мастер работают на русском; англоязычные спеки замедляют ревью
и увеличивают расхождение при реализации.

## Технологии и ограничения безопасности

- **Frontend**: React 18, TypeScript, Vite, SCSS modules.
- **Backend**: Vercel Serverless Functions, единая точка входа `api/index.js` (CommonJS).
- **Данные**: Supabase Postgres + Storage; схема в `supabase/schema.sql`; seed из
  `data/fabrics.json` через `npm run seed:supabase`.
- **Деплой**: Vercel CI из `main`; переменные окружения — по README.
- **Дизайн-система**: `PRODUCT.md`, `DESIGN.md`, skill Impeccable — новый UI ДОЛЖЕН
  соответствовать тёплому craft-бренду (без generic AI-градиентов).
- **Доступность**: контраст WCAG 2.1 AA; `prefers-reduced-motion`; комфортные touch-зоны
  на мобильных.

## Процесс разработки

1. **Specify** → **Plan** → **Tasks** через команды Spec Kit; gates Constitution Check в
   `plan.md` ДОЛЖНЫ пройти до реализации.
2. Локально: `npm run dev:mock` для UI; `npm run dev` с `.env` — auth, API, Supabase.
3. Каждый PR ДОЛЖЕН проверять: mock-режим работает, live API не сломан или версионирован,
   секретов в diff нет, `CHECKLIST.md` обновлён при закрытии пунктов roadmap.
4. Визуальные изменения ЖЕЛАТЕЛЬНО сверять с `DESIGN.md` и принципами `PRODUCT.md`.
5. Ломающие изменения API или формата данных ДОЛЖНЫ обновлять таблицу API в `README.md`
   и mock-фикстуры в том же changeset.

## Управление (Governance)

Конституция имеет приоритет над ad-hoc практиками для фич, планируемых через Spec Kit. При
конфликте принципы конституции важнее удобства.

**Процедура изменений**:

1. Предложить правки через `/speckit-constitution` с обоснованием и типом bump версии.
2. Распространить изменения на `.specify/templates/*` и затронутые guidance-доки.
3. Зафиксировать версию, дату ратификации и дату последней правки в этом файле.

**Политика версий**: semver — MAJOR при удалении/переопределении принципов, MINOR при
новых принципах или существенных расширениях, PATCH при уточнениях и опечатках.

**Соответствие**: секция Constitution Check в каждом `plan.md` ДОЛЖНА быть заполнена.
Нарушения требуют явного обоснования в Complexity Tracking. Операционные гайды:
`README.md`, `PRODUCT.md`, `DESIGN.md`, `CHECKLIST.md`, `docs/ADMIN.md`.
