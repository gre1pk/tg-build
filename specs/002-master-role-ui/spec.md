## Пользовательские сценарии и тестирование *(обязательно)*

### User Story 1 — Мастер открывает админку из приложения (Приоритет: P1)

Мастер студии открывает Mini App из Telegram. На клиентских экранах видит кнопку «Панель мастера», ведущую на `/#/admin` с полным меню (заявки, ткани, портфолио). Обычные клиенты кнопки **не** видят. Мастер управляет контентом и заявками без ручного ввода URL.

**Почему этот приоритет**: вход в админку сейчас спрятан; мастер — основной пользователь панели.

**Независимый тест**: Telegram ID в `MASTER_TELEGRAM_IDS` → кнопка видна → `/admin` → CRUD тканей и заявок работает.

**Сценарии приёмки**:

1. **Дано** пользователь с `role=master`, **Когда** открывает главную, **Тогда** видит кнопку «Панель мастера» и попадает на `/#/admin`.
2. **Дано** `role=master`, **Когда** открывает `/admin/fabrics` и создаёт ткань, **Тогда** операция успешна (как у admin).
3. **Дано** обычный клиент без роли, **Когда** открывает приложение, **Тогда** staff-кнопки нет; `/admin` → «Нет доступа».
4. **Дано** ID только в `ADMIN_TELEGRAM_IDS`, **Когда** проверка роли, **Тогда** `role=admin`, те же права на API и UI.

---

### User Story 2 — Админ (технический) с теми же правами (Приоритет: P2)

Пользователь с ролью **admin** видит кнопку «Админка» (или эквivalent) и полный доступ к `/admin/*` и `/api/admin/*`. Поведение совпадает с master; списки env могут различаться (например, разработчик только в `ADMIN_TELEGRAM_IDS`).

**Почему этот приоритет**: сохранить текущий `ADMIN_TELEGRAM_IDS` и не сломать существующий доступ.

**Независимый тест**: ID в `ADMIN_TELEGRAM_IDS` → `role=admin`, `isStaff=true`, полный CRUD.

**Сценарии приёмки**:

1. **Дано** `role=admin`, **Когда** POST `/api/admin/fabrics`, **Тогда** 201.
2. **Дано** ID в обоих env-списках, **Когда** resolve роли, **Тогда** `role=admin` (приоритет admin над master в ответе API).
3. **Дано** существующие smoke админки (orders, fabrics, portfolio), **Когда** пользователь master, **Тогда** те же результаты, что у admin.

---

### User Story 3 — Полировка клиентского UI (Приоритет: P3)

Клиентские экраны выгляят аккуратнее и единообразнее: отступы, типографика, кнопки и карточки по `DESIGN.md`; читаемость в light/dark теме Telegram; без регрессии потоков каталог → заявка.

**Почему этот приоритет**: доверие и конверсия; не блокирует роли.

**Независимый тест**: визуальный проход + `npm run build`.

**Сценарии приёмки**:

1. **Дано** light theme, **Когда** главная и каталог, **Тогда** контраст и CTA по `DESIGN.md`.
2. **Дано** dark theme, **Тогда** нет «белых дыр»; `--tg-theme-*` где уместно.
3. **Дано** полировка, **Когда** smoke клиентского flow, **Тогда** без регрессий.

---

### Граничные случаи

- Пустые `MASTER_TELEGRAM_IDS` и `ADMIN_TELEGRAM_IDS` → `role=null`; staff-кнопки нет; `/admin/*` → 403.
- Только `MASTER_TELEGRAM_IDS` → полный доступ staff (не урезанный).
- Только `ADMIN_TELEGRAM_IDS` → как сейчас, плюс явная роль `admin` в API.
- JWT истёк → staff-кнопка скрыта; не показывать админку без валидной сессии.
- `MASTER_TELEGRAM_CHAT_ID` (bot notify) — отдельная переменная; не путать с `MASTER_TELEGRAM_IDS`.

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: Сервер ДОЛЖЕН определять роль: `admin` если ID ∈ `ADMIN_TELEGRAM_IDS`; иначе `master` если ID ∈ `MASTER_TELEGRAM_IDS`; иначе `null`. Оба списка — server-only, comma-separated numeric IDs.
- **FR-002**: `GET /api/admin/me` ДОЛЖЕН возвращать `{ role: 'admin' | 'master' | null, isStaff: boolean, user }`. `isStaff === true` для `admin` и `master`. Поле `isAdmin` MAY дублировать `isStaff` для обратной совместимости (deprecated в пользу `isStaff`).
- **FR-003**: **Все** маршруты `/api/admin/*` (orders, fabrics, portfolio, upload) ДОЛЖНЫ быть доступны и `master`, и `admin` — одинаковые права на текущем этапе.
- **FR-004**: Клиент ДОЛЖЕН показывать staff-кнопку **только** при `isStaff`. Подпись: «Панель мастера» при `role=master`, «Админка» при `role=admin` (или единая — уточнить на clarify).
- **FR-005**: Staff-кнопка ДОЛЖНА вести на `/#/admin` (полное меню) для обеих ролей.
- **FR-006**: `AdminGate` ДОЛЖЕН пропускать `master` и `admin` на все страницы `/admin/*` без скрытия разделов.
- **FR-007**: Env `MASTER_TELEGRAM_IDS` задокументирован в `.env.example`, README, `docs/ADMIN.md`. Отличие от `MASTER_TELEGRAM_CHAT_ID` явно описано.
- **FR-008**: Полировка UI клиентских страниц (`/`, `/fabrics`, `/fabrics/:id`, `/order`) и шапки админки по `DESIGN.md`; без новых npm UI-библиотек.

### Ключевые сущности

- **UserRole**: `admin` | `master` | `null` — метка для UI и аудита; **не** matriz прав в v1.
- **Staff**: объединение `master` ∪ `admin` для проверки доступа к `/admin` и `/api/admin/*`.
- **StaffEntryButton**: условная кнопка входа для staff.

## Критерии успеха *(обязательно)*

### Измеримые результаты

- **SC-001**: Staff открывает админку за ≤ 2 тапа с главной (кнопка → `/admin`).
- **SC-002**: Master и admin проходят один и тот же smoke CRUD (fabrics, portfolio, orders).
- **SC-003**: Обычный клиент не видит staff-кнопку на главной, каталоге и заявке.
- **SC-004**: `npm run build` и клиентский smoke без регрессий после UI polish.

## Допущения

- Master и admin — равные права **сейчас**; разные env-списки для удобства (мастер vs разработчик) и разные подписи в UI.
- Один человек может быть в обоих списках → в API `role=admin`.
- Роли только через env; таблица `users` — **вне scope**.
- Hero/реальные фото бренда — отдельная контент-задача; здесь layout/typography polish.
