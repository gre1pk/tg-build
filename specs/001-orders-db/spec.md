## Пользовательские сценарии и тестирование *(обязательно)*

### User Story 1 — Клиент отправляет заявку (Приоритет: P1)

Клиент на странице `/order` прикрепляет фото мебели (или пишет комментарий), опционально указывает выбранную ткань и нажимает «Отправить». Заявка сохраняется в Supabase. Клиент видит экран успеха «Заявка сохранена» — **без** автоматического открытия Telegram или Share API. Мастер обрабатывает заявку через `/admin/orders`.

**Почему этот приоритет**: без сохранения мастер не видит историю заявок; текущий flow только открывает Telegram.

**Независимый тест**: POST `/api/orders` с JWT → запись в таблице `orders`; клиент видит подтверждение «Заявка сохранена».

**Сценарии приёмки**:

1. **Дано** авторизованный пользователь с фото и комментарием, **Когда** отправляет форму, **Тогда** создаётся заявка со статусом `new`, фото в Storage, текст комментария в БД.
2. **Дано** заявка без фото, но с комментарием, **Когда** отправка, **Тогда** заявка создаётся (фото URL nullable).
3. **Дано** выбранная ткань (`fabricId` в query), **Когда** отправка, **Тогда** в заявке сохраняются `fabric_id` и снимок названия ткани.
4. **Дано** заявка с фото без комментария, **Когда** отправка, **Тогда** заявка создаётся (`comment` nullable).
5. **Дано** нет ни фото, ни комментария (даже при `fabricId`), **Когда** отправка, **Тогда** 400; клиент показывает ошибку валидации.

---

### User Story 2 — Мастер видит новые заявки (Приоритет: P2)

Мастер в админке (`/admin/orders`) по умолчанию видит **активные** заявки (`new`, `in_progress`): дата, клиент (имя / Telegram ID), фото, комментарий, ткань, статус. Завершённые и отклонённые (`done`, `cancelled`) — во вкладке или фильтре «Архив». Может менять статус только **вперёд**: `new` → `in_progress` → `done`, либо отклонить → `cancelled` (из `new` или `in_progress`). Из `done` и `cancelled` переходов нет.

**Почему этот приоритет**: ценность для мастера — не терять заявки и вести простой учёт.

**Независимый тест**: админ с `ADMIN_TELEGRAM_IDS` открывает `/admin/orders`, видит заявку из US1, меняет статус.

**Сценарии приёмки**:

1. **Дано** заявка со статусом `new`, **Когда** мастер ставит `in_progress`, **Тогда** статус обновляется в БД и в списке.
2. **Дано** заявка со статусом `new` или `in_progress`, **Когда** мастер ставит `cancelled`, **Тогда** статус обновляется; дальнейшие переходы недоступны.
3. **Дано** заявка со статусом `done` или `cancelled`, **Когда** PATCH с другим статусом, **Тогда** 400.
4. **Дано** заявки со статусами `done` и `cancelled`, **Когда** мастер открывает `/admin/orders`, **Тогда** они не видны в списке по умолчанию; доступны во вкладке «Архив».
5. **Дано** не-админ, **Когда** запрос к `/api/admin/orders`, **Тогда** 403.

---

### User Story 3 — Уведомление мастера в Telegram (Приоритет: P3)

При новой заявке бот отправляет мастеру в Telegram (chat_id из `MASTER_TELEGRAM_CHAT_ID`) краткое сообщение: **имя клиента**, **начало комментария** (до 100 символов; если комментария нет — строка «Без комментария»), **ссылка** на `/admin/orders`. Фото и Telegram ID в сообщение не включаются.

**Почему этот приоритет**: ускоряет реакцию, но MVP работает и без бота (мастер смотрит админку).

**Независимый тест**: после POST `/api/orders` мастер получает сообщение от бота (если `MASTER_TELEGRAM_CHAT_ID` задан).

**Сценарии приёмки**:

1. **Дано** настроен `TELEGRAM_BOT_TOKEN` и chat id мастера, **Когда** новая заявка с комментарием, **Тогда** бот отправляет сообщение с именем клиента, первыми ≤100 символами комментария и ссылкой на админку (ошибка отправки не откатывает сохранение заявки).
2. **Дано** заявка только с фото, без комментария, **Когда** новая заявка, **Тогда** в сообщении вместо комментария — «Без комментария».

---

### Граничные случаи

- Нет JWT / истёкшая сессия → 401, форма показывает ошибку авторизации.
- Нет фото и нет комментария → 400 (сервер и клиент); только `fabricId` недостаточно.
- Фото > 10 МБ или не image → 400 до записи в Storage.
- Supabase недоступен → 500, клиент показывает «Не удалось сохранить заявку»; **fallback в Telegram не предусмотрен** (мастер узнаёт о заявке только через админку после успешного сохранения).
- Дубль отправки (double tap) → идемпотентность не обязательна в v1; допустимы две записи.
- Недопустимый переход статуса (например `done` → `in_progress`, `cancelled` → `new`) → 400.
- Уведомление ботом упало → заявка всё равно в БД, в логах ошибка.

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: `POST /api/orders` (JWT) создаёт заявку из JWT-профиля: `telegram_id`, `user_first_name`, `user_username` (nullable), `comment` (nullable), `fabric_id` (nullable), `fabric_snapshot` (nullable), `photo_url` (nullable), `status=new`. **Валидация**: обязательно хотя бы одно — непустой `comment` или `photo`; иначе 400.
- **FR-002**: Фото заявки загружается в Supabase Storage bucket `order-images` через сервер (base64 или multipart; переиспользовать паттерн `/api/admin/upload`).
- **FR-003**: `GET /api/admin/orders` — список заявок для админа, сортировка по `created_at desc`; опциональный query `status` (один статус или через запятую) для фильтрации на сервере.
- **FR-004**: `PATCH /api/admin/orders/:id` — обновление статуса с валидацией переходов: `new` → `in_progress` | `cancelled`; `in_progress` → `done` | `cancelled`; `done` и `cancelled` — терминальные (400 при смене).
- **FR-005**: Страница `/admin/orders` — вкладка «Активные» (`new`, `in_progress`) по умолчанию; «Архив» (`done`, `cancelled`); смена статуса (включая «Отклонить» → `cancelled`).
- **FR-006**: `OrderRequestPage` после успешного API показывает «Заявка сохранена»; **не** вызывает Share API и **не** открывает чат с мастером (мастер работает через админку). До отправки проверяет то же правило, что FR-001 (фото или комментарий).
- **FR-007**: (P3) После успешного `POST /api/orders` сервер вызывает Bot API `sendMessage` в `MASTER_TELEGRAM_CHAT_ID`. Текст: имя клиента, комментарий (truncate до 100 символов или «Без комментария»), URL админки `/admin/orders`. Сбой отправки — только log, без отката заявки.
- **FR-008**: (P4) `DELETE /api/admin/orders/:id/photo` (admin) — только для `status` ∈ `done`, `cancelled` и при наличии `photo_url`. Удаляет файл из bucket `order-images`, ставит `photo_url = null`. Строка заявки сохраняется. UI: кнопка «Удалить фото» во вкладке «Архив» с confirm.

### Ключевые сущности

- **Order**: заявка клиента — связь с пользователем Telegram, текст, фото, ткань, статус, timestamps.
- **OrderStatus**: enum `new`, `in_progress`, `done`, `cancelled`. Разрешённые переходы: `new` → `in_progress` | `cancelled`; `in_progress` → `done` | `cancelled`.

## Критерии успеха *(обязательно)*

### Измеримые результаты

- **SC-001**: Клиент отправляет заявку с фото за ≤ 3 минут (без регрессии UX).
- **SC-002**: 100% успешных POST сохраняют строку в `orders` (проверка smoke на staging).
- **SC-003**: Мастер меняет статус за ≤ 2 клика в админке.
- **SC-004**: (P3) Уведомление доставляется в течение 30 с после создания заявки.
